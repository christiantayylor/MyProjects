## A. Cloud Deployment -        Render.com

The Vacation Planner application is deployed using **Render.com**, a cloud platform that hosts both the Express backend API and the React frontend as two separate services.

- **Backend (API):** `https://vacation-planner-api.onrender.com`
- **Frontend:** `https://vacation-planner-client.onrender.com`

---

## A.1 Justification for Selecting Render.com

Render.com was selected as the cloud deployment provider for the Vacation Planner application for the following reasons:

**Cost - Free tier available.** Render offers a fully functional free tier for both web services (Node.js backend) and static sites (React frontend). No credit card is required to deploy on the free plan, which directly aligns with the capstone guidance to use free-of-charge deployment options.

**Native support for Node.js and Docker.** The Vacation Planner backend is built on Node.js with Express. Render supports Node.js web services natively and also supports Docker container deployments, satisfying the A.2 container image requirement without requiring a separate container registry.

**Static site hosting for React.** Render's Static Site service automatically runs `npm run build` on the React frontend and serves the compiled output from a global CDN, which is the correct deployment model for a Create React App project.

**GitLab integration.** Render connects directly to GitLab repositories and can automatically redeploy whenever a new commit is pushed to the selected branch, enabling continuous deployment with no manual steps after initial setup.

**Simplicity of configuration.** Render supports an infrastructure-as-code file (`render.yaml`) placed in the root of the repository, which defines both the backend and frontend services, their environment variables, and build commands in a single declarative file. This makes the deployment reproducible and auditable.

**No server management required.** Render is a fully managed platform-as-a-service (PaaS). There is no need to configure virtual machines, manage operating system updates, or handle networking - allowing the deployment to be completed entirely through a web interface and a configuration file.

## A.2 Container Image Implementation

The backend Express server is containerized using **Docker**. The container image is defined in `server/Dockerfile` and is used by Render's Docker-based web service to build and run the application in an isolated, reproducible environment.

### Dockerfile (`server/Dockerfile`)

```dockerfile
FROM node:18-alpine

WORKDIR /app

COPY package.json ./
RUN npm install --production

COPY . .

RUN mkdir -p /app/database

RUN node src/scripts/initDb.js

EXPOSE 5000

CMD ["node", "src/server.js"]
```

### How the Container Image Works

| Instruction | Purpose |
|---|---|
| `FROM node:18-alpine` | Uses the official Node.js 18 image on Alpine Linux — a minimal base image that keeps the container size small |
| `WORKDIR /app` | Sets the working directory inside the container to `/app` |
| `COPY package.json ./` | Copies only the package manifest first, allowing Docker to cache the dependency layer |
| `RUN npm install --production` | Installs only production dependencies, excluding dev tools from the image |
| `COPY . .` | Copies all remaining server source files into the container |
| `RUN mkdir -p /app/database` | Creates the database directory so SQLite has a location to write the file |
| `RUN node src/scripts/initDb.js` | Initializes the SQLite database tables at image build time |
| `EXPOSE 5000` | Documents that the container listens on port 5000 |
| `CMD ["node", "src/server.js"]` | Defines the command that runs when the container starts |

The container image is built automatically by Render when a new commit is pushed to the GitLab repository. Render pulls the repository, builds the Docker image using the Dockerfile, and deploys the resulting container to its infrastructure.
