## 1. Introduction

Vacation Planner is a full-stack web application that enables users to plan, organize, and track all aspects of a trip in one place. Users can create trips, search and book hotels and flights via the Booking.com API, discover excursions, track all associated costs through an expense reporting system, and edit or delete trip data at any time.

This document describes the software architecture, class hierarchy, component relationships, data flow, and design decisions that underpin the application.

---

## 2. System Architecture Overview

The application follows a three-tier architecture: a React.js frontend (client), a Node.js/Express REST API (server), and a SQLite database. The client communicates exclusively with the Express server via HTTP. The server handles all third-party API calls so API keys are never exposed to the browser.

| Tier 1: Client | Tier 2: Server | Tier 3: Data |
|---|---|---|
| React.js SPA | Express REST API | SQLite Database |
| Pages, Components, api.js service layer | Route modules, middleware, third-party API proxy | Trips, Bookings, Excursions, Expenses, Users |

## 3. Class Diagram

The class diagram below shows the OOP model layer in `server/src/models/`. `TripItem` is the base class providing shared properties and a polymorphic `getSummary()` method. `Booking` and `Excursion` each extend `TripItem`, overriding `getSummary()` to produce type-specific output. This satisfies the capstone requirements for inheritance, polymorphism, and encapsulation.

### Base Class — `TripItem`

```
┌─────────────────────────────┐
│          TripItem           │
├─────────────────────────────┤
│  # tripId : integer         │
│  # cost   : float           │
│  # date   : string          │
├─────────────────────────────┤
│  + constructor(tripId, cost,│ 
│    date)                    │
│  + getSummary() : string    │
└─────────────────────────────┘
               ▲
       ┌───────┴────────┐
       │                │
┌──────┴──────┐  ┌──────┴──────┐
│   Booking   │  │  Excursion  │
├─────────────┤  ├─────────────┤
│ - type      │  │ - name      │
│ - details   │  │ - description│
├─────────────┤  ├─────────────┤
│ + constructor(tripId, cost,  │  │ + constructor(tripId, cost,  │
│   date, type, details)       │  │   date, name, desc)          │
│ + getSummary() {override}    │  │ + getSummary() {override}    │
└─────────────┘  └─────────────┘
```

- **Inheritance:** `Booking extends TripItem` | `Excursion extends TripItem`
- **Polymorphism:** `getSummary()` is overridden in each subclass. Route handlers can call `item.getSummary()` on any `TripItem` without knowing the concrete type.
- **Encapsulation:** Fields use access modifiers (`#` protected, `-` private, `+` public). State and behavior are fully encapsulated within each class.

## 6. Database Schema

All tables are created via `server/src/scripts/initDb.js` on first run. The database file is stored at `server/database/vacation.db`.

| Table | Key Columns | Types | Notes |
|---|---|---|---|
| trips | id, userId, destination, startDate, endDate, createdAt | INT / TEXT | Root entity. Cascade-deletes all related rows on deletion. |
| bookings | id, tripId, type, details, cost, bookingDate | INT / TEXT / REAL | details column stores hotel or flight info as a JSON string. |
| excursions | id, tripId, name, description, date, cost | INT / TEXT / REAL | Populated from Booking.com attractions API. |
| expenses | id, tripId, category, amount, note, date | INT / TEXT / REAL | Manual entries. date defaults to CURRENT_TIMESTAMP. |
| users | id, name, email, createdAt | INT / TEXT | Single default user (userId = 1) used throughout. |

## 7. REST API Endpoints

All endpoints are prefixed with `/api` and served on port `5000`.

| Method | Endpoint | Description |
|---|---|---|
| GET | /api/trips | Return all trips |
| GET | /api/trips/:id | Return a single trip by ID |
| POST | /api/trips | Create a new trip |
| PUT | /api/trips/:id | Update trip destination and dates |
| DELETE | /api/trips/:id | Delete trip and cascade-delete related data |
| GET | /api/bookings/:tripId | Return all bookings for a trip |
| POST | /api/bookings | Create a hotel or transportation booking |
| DELETE | /api/bookings/:id | Delete a booking |
| GET | /api/excursions/search | Search Booking.com attractions by city + dates |
| GET | /api/excursions/destid | Resolve a city name to a Booking.com cityUfi |
| GET | /api/excursions/:tripId | Return saved excursions for a trip |
| POST | /api/excursions | Save an excursion to a trip |
| GET | /api/expenses/:tripId | Return all expenses for a trip |
| POST | /api/expenses | Add a manual expense entry |
| GET | /api/hotels/search | Search hotels via Booking.com RapidAPI |
| GET | /api/flights/transportation/search | Search flights via Flight Fare Search RapidAPI |
| GET | /api/cities | City autocomplete via GeoDB RapidAPI (cached, rate-limited) |

## 8. Security Features

- Parameterized SQL queries throughout all server routes prevent SQL injection.
- API keys stored in `server/.env` and loaded via dotenv — never transmitted to the browser.
- CORS restricted to `http://localhost:3000` in `server.js`, blocking unauthorized origins.
- All third-party API calls proxied through Express — credentials stay server-side.
- Client-side validation (required fields, date order checks) and server-side validation (HTTP 400 on missing fields) at every data entry point.
- In-memory caching + 1.1s rate limiter on the `/api/cities` route prevent GeoDB API quota exhaustion.

---

## 9. Scalability Design

- **Modular route architecture** — each domain is an independent Express router. New features require only a new route file and one line in `routes/index.js`.
- **Service layer abstraction** — all HTTP calls centralized in `client/src/services/api.js`. Changing the base URL or auth headers requires editing one file.
- **Database abstraction** — all DB access through `config/database.js`. Swapping SQLite for PostgreSQL requires changing only the connection module.
- **In-memory caching** on excursion and hotel routes reduces redundant external API calls under repeated queries.
- **Environment-based configuration** via `.env` — the app can move from development to production without code changes.
- **concurrently package** allows single-command startup (`npm start` from root) for consistent developer experience.
